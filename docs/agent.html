0. 실행 요약

Executive Summary

핵심 결론

  • 본질 급여 프로젝트의 핵심은 코어(정확성/권한/감사)입니다.
  • Agent 에이전트는 업무 인터페이스·자동화 레이어로 도입합니다.
  • 순서 Read-only → Write(승인)로 단계적 확장합니다.

MVP 원칙

  • 예외는 “자동처리”보다 탐지+승인 중심
  • RBAC/감사로그/마감 상태는 MVP에도 포함
  • 정확성은 골든 케이스 회귀테스트로 잠금

결과물(문서/산출물)

  • 규칙 결정표(산식/예외/한도)
  • 데이터 계약(입/출력 스키마)
  • 승인/마감 상태머신
  • 권한 매트릭스(RBAC) + 감사로그
리스크 안내
급여는 금전 및 민감정보가 포함된 운영 영역입니다. “속도”를 위해 코어 통제(승인/권한/감사/테스트)를 생략하면, 운영 장애 및 감사 이슈로 연결될 수 있습니다.

1. 아키텍처(2-Layer) 인포그래픽

Infographic

2-Layer 분리의 목적

  • 코어: 정확성/권한/감사/테스트(절대 안정)
  • Agent: 업무 인터페이스/자동화(확장 가능)
  • 양쪽 변경 속도를 분리하여 운영 안정성 확보
인포그래픽: 코어(정확성·통제)와 Agent(자동화·UI)를 분리하여 개발 속도와 운영 안정성을 동시에 확보합니다.

2. 추진 단계(스프린트) 타임라인

Timeline
0

Sprint 0 (1주): 결정표/데이터 계약/MVP 범위 확정

“무엇을 만들지”를 잠그는 단계. 이후 재작업 비용을 급격히 줄입니다.

  • 급여 규칙 결정표(산식/예외/한도)
  • 입력/출력 스키마 + 엑셀 템플릿
  • 권한(RBAC) + 감사로그 요구사항
1

Sprint 1 (2주): 단일 시나리오 E2E 구축

입력→검증→계산→명세서 출력까지 “흐름이 끊기지 않게” 만듭니다.

  • 골든 케이스 20~50개 구성
  • 기본 검증/예외 리스트업
  • 명세서/기본 리포트
2

Sprint 2 (2주): 마감(확정)/승인/감사/소급

급여 시스템의 핵심인 “통제”를 완성하는 단계입니다.

  • 승인 흐름 + 마감 상태머신
  • 감사로그(전/후, 사유, 승인자)
  • 소급 정산 회귀테스트
3

Sprint 3 (2주): 전표/내보내기 + Read-only Agent

현업 가치가 크게 체감되는 “분석/요약”을 도입합니다.

  • 전표/CSV/XLSX 내보내기
  • 증감 원인 요약, 예외 설명
  • 근거(출처/계산 근거) 표기
추가: Sprint 4(확장) 권장 범위
  • 근태/ERP 연동, 자동 동기화
  • Write Agent(초안/작업 실행) + 승인 워크플로우
  • 감사 대응용 증빙 패키지 자동 생성

5. AI Agent 설계(읽기 → 쓰기)

Read → Write

Phase 1 · Read-only Agent (권장)

  • 증감 분석: “전월 대비 증가 원인 Top 3”
  • 예외 설명: 누락/중복/이상치 후보 해석
  • 규정 Q&A: 근거(문서/규정 링크) 포함
  • 리포트 요약: 부서/직급/프로젝트별 집계

Phase 2 · Write Agent (승인 포함)

  • 초안 생성: 신규 입사자 급여항목 기본값(임시저장)
  • 정형 작업: 체크리스트 실행/알림 발송
  • 필수: “제안 → 검토 → 승인 → 반영” + 감사로그
  • 금지: 승인 없이 확정/지급파일/전표 확정
Write Agent는 “반영 권한”이 아니라 “작업 보조/제안” 중심으로 설계하고, DB 반영은 승인(4-eyes)과 롤백을 전제로 운영하십시오.
권장: 에이전트 권한 레벨(L0~L3)
레벨 허용 기능 통제/기록
L0(조회) 조회/요약/질의응답 조회 로그(사용자/질의/결과 요약) 기록
L1(초안) 입력 초안 생성(저장 전/임시저장) 최종 반영은 승인 필요
L2(실행) 정형 작업 실행(알림/체크리스트) 실행 로그 + 결과 리포트 보관
L3(반영) DB 반영(수정/갱신) 승인(4-eyes) + 감사로그 + 롤백

6. 테스트·품질(DoD)

Quality

골든 케이스 기반 회귀 테스트

  • 대표 직원군 정답셋 구축(정규/계약/입·퇴사/휴직 등)
  • 입력 → 계산 → 명세서/전표 출력까지 E2E 검증
  • 규칙 변경 시 과거 월 재현 가능(버전 고정)
# 권장 테스트 축
- 근태 누락/중복
- 연장/야간/휴일 수당
- 소급(전월 정정) 및 차액 정산
- 입사/퇴사 월 일할 계산
- 공제 항목 한도/상한/하한
- 마감 후 변경 제한 및 재오픈

완료 기준(DoD) 제안

  • 정확성: 골든 케이스 일치율 목표 합의(예: 95~99%)
  • 통제: RBAC 시나리오 테스트 통과
  • 감사: 변경 이력(전/후, 사유, 승인자) 추적 가능
  • 안정성: 대량 처리 성능 기준 합의

7. 보안·운영(핵심 통제)

Controls

민감정보(PII) 처리

  • 최소 수집/최소 노출(화면·로그 마스킹)
  • 접근제어(역할/범위/감사 역할 분리)
  • 전송 구간 TLS, 저장 시 암호화(가능 시)
  • 계좌/식별정보는 토큰화/부분 마스킹 우선

운영 프로세스

  • 개발/스테이징/운영 환경 분리
  • 월 마감 체크리스트(검증→검토→확정→내보내기)
  • 규칙 변경: 요청→검토→테스트→릴리즈→적용 월 지정
  • 감사 대응: “직원/월” 증빙 패키지 자동 생성(확장)
급여 데이터가 외부 서비스로 전송되는 구조는 법적/계약적 검토가 필요합니다. 착수 시 데이터 처리 범위를 문서로 확정하십시오.
