핵심 가치

  • AI 매칭: 사진 스타일 분석으로 최적의 프리랜서 추천

  • 신뢰: 리뷰 및 포트폴리오 검증
  • 편리성: 크레딧 + 구독 하이브리드 결제

0.4. Next.js 웹의 SEO 최적화

Next.js 웹은 검색 엔진 노출을 위해 사용됩니다.

중요: 왜 웹이 필요한가?

Flutter 앱은 앱스토어를 통해서만 다운로드 가능합니다. 하지만 사용자들이 구글에서 "서울 웨딩 사진사"를 검색할 때, 우리 프리랜서들이 검색 결과에 나타나야 합니다. 이를 위해 Next.js로 공개 웹 페이지를 만듭니다.

SEO가 필요한 페이지 (Next.js 전용)

페이지 목적 SEO 키워드 Flutter 앱 연계
랜딩 페이지 신규 고객 유입 "웨딩 프리랜서", "스냅 사진사" 앱 다운로드 유도
프리랜서 프로필 프리랜서 홍보 "서울 웨딩 사진사", "부산 메이크업" 앱에서 상세 정보 + 채팅
블로그 콘텐츠 마케팅 "결혼 준비", "웨딩 팁" 앱 다운로드 CTA

실제 사용 시나리오

  1. 사용자가 구글 검색: "서울 웨딩 사진사"
  2. Next.js 프로필 페이지 노출: 김철수 작가님 프로필이 검색 결과에 표시
  3. 웹에서 기본 정보 확인: 포트폴리오, 가격대, 리뷰
  4. 앱 다운로드 유도: "채팅으로 견적 요청하기" 버튼 클릭
  5. Flutter 앱에서 실제 거래: 채팅, 결제, 예약

  • 신뢰 관계: 코드 변경에 대한 충분한 이해와 책임감이 있을 때

⚠️ 주의: Write 권한 부여 시에도 브랜치 보호 규칙으로 main 브랜치 직접 push는 차단됩니다.

1. 요약

- 종료 시각: 2025-01-01 16:45 (2시간 13분)
- 영향: 전체 사용자, 로그인 불가
- 등급: P0 (Critical)

2. 타임라인
14:32 - Crashlytics 알림 감지
14:35 - 팀 알림, 조사 시작
14:50 - 원인 파악 (null 참조)
15:20 - 수정 완료, 빌드 시작
15:45 - 앱스토어 제출
16:45 - 배포 완료, 장애 종료

3. 근본 원인 (Root Cause)
- User 모델의 name 필드가 nullable로 변경됨
- LoginScreen에서 user.name에 ! 연산자 사용
- name이 null인 사용자 로그인 시 크래시

4. 해결 방법
- null-safe 처리 추가
- user.name ?? '익명' 으로 변경

5. 왜 놓쳤는가?
- API 스펙 변경 시 팀 공유 부족
- nullable 필드에 대한 충분한 테스트 부족
- 코드 리뷰에서 발견 못함

6. 재발 방지 대책
[ ] User 모델 변경 시 팀 Slack 알림
[ ] nullable 필드는 모두 null-safe 처리 규칙 문서화
[ ] 통합 테스트에 "null 케이스" 추가
[ ] 코드 리뷰 체크리스트에 "nullable 처리" 항목 추가

- API 변경은 Breaking Change로 간주해야 함
- 테스트 케이스에 edge case 더 많이 필요
- Feature Flag로 점진적 출시 고려

9.5. Supabase RLS (Row Level Security)

-- 사용자는 자신의 데이터만 조회 가능
CREATE POLICY "Users can view own data"
ON profiles FOR SELECT
USING (auth.uid() = user_id);

CREATE POLICY "Users can update own data"
ON profiles FOR UPDATE
USING (auth.uid() = user_id);

-- 채팅: 참여자만 메시지 조회
CREATE POLICY "Users can view own messages"
ON messages FOR SELECT
USING (
  auth.uid() IN (
    SELECT user_id FROM chat_participants
    WHERE chat_id = messages.chat_id
  )
);

CD (Continuous Deployment): 테스트 통과 시 자동 배포

CI/CD 파이프라인
1
코드 Push / PR 생성
GitHub에 코드 변경사항 업로드
↓
2
자동 테스트 실행 (CI)
flutter analyze • flutter test • 코드 커버리지
↓
3
코드 리뷰 및 승인
최소 1명 승인 필요
↓
4
자동 배포 (CD)
Flutter: TestFlight / Firebase App Distribution
Next.js: Vercel Preview (PR) → Production (main)
FastAPI: Railway 자동 배포

우리 프로젝트의 CI/CD

  • PR 생성 시 자동 테스트 실행
  • main 브랜치 Merge 시 자동 배포
  • Flutter: TestFlight/Firebase App Distribution
  • Next.js: Vercel 자동 배포
  • FastAPI: Railway 자동 배포

11.2. GitHub Actions 워크플로우 - Flutter

# .github/workflows/flutter-ci.yml
name: Flutter CI

on:
  pull_request:
    branches: [develop, main]
    paths:
      - 'apps/flutter/**'
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      
      - name: Flutter 설정
        uses: subosito/flutter-action@v2
        with:
          flutter-version: '3.19.0'
          channel: 'stable'
      
      - name: 의존성 설치
        working-directory: apps/flutter
        run: flutter pub get
      
      - name: 코드 분석
        working-directory: apps/flutter
        run: flutter analyze
      
      - name: 테스트 실행
        working-directory: apps/flutter
        run: flutter test --coverage
      
      - name: 커버리지 업로드
        uses: codecov/codecov-action@v3
        with:
          files: apps/flutter/coverage/lcov.info
